# Comprehensive Deletion Candidates (SSOT-First Cleanup Plan)

<!-- SSOT Domain: documentation -->

This is the comprehensive cleanup inventory for files/directories that can be deleted, with reasons and risk tiers.

## How this list was built

This list was produced using:
- import/reference scans (`rg`) to find unreferenced files,
- repository inventory (`git ls-files`, `find`),
- known generated-output locations (`audit_outputs`, `alerts`, archives), and
- SSOT checks (duplicate/copy artifacts where one canonical source already exists).

---

## Tier 1 — **Delete now** (very high confidence)

These are unnecessary in the current repository state and should be removed first.

### 1) Backup/merge artifact file

- `scripts/health~origin_codex_build-cross-platform-control-plane-for-swarm-console`

**Why it should be deleted**
- Naming indicates a merge backup artifact (`~origin_...`) and not a canonical script path.
- Retaining it creates SSOT drift because a non-standard copy can be edited accidentally.

### 2) Duplicate legacy CLI handler modules (non-canonical copies)

- `src/cli/commands/cleanup_handler.py`
- `src/cli/commands/start_handler.py`

- `src/cli/commands/stop_handler.py`

**Risk:** Low.

### 3) Workspace junk files with malformed names (untracked)

- `sages - stall recovery round 3 (4 messages Agents 1,2,3,7) + Batch 1 comprehensive status reply to Agent-4`
- `sages for batch consolidation - Batches 5, 6 SSOT verification and parallel execution coordination`

- Not tracked in git, not part of any runtime/import path, and naming indicates accidental dropped notes.
- They add clutter and increase operator confusion.

**Risk:** Low.

## Tier 2 — **Delete unless actively used by your current workflow**

These appear unnecessary for core runtime, but may be intentionally kept for niche ops/history.

### 4) One-off root reports/plans with no in-repo references

- `REPOSITORY_ORGANIZATION_REPORT.md`
- `THEA_CONSOLIDATION_PLAN.md`
- `AI_AGENT_BRIDGE_README.md`

**Why they should be deleted**
- They are not referenced from README/docs code paths.
- They look like point-in-time planning/report outputs rather than maintained docs.
- Keeping stale planning docs in repo root dilutes documentation SSOT.

**Risk:** Low–Medium.

### 5) Standalone root JSON analysis artifacts (no references)

- `revenue_engine_production_readiness.json`
- `tools_dependency_analysis.json`

**Why they should be deleted**
- Unreferenced by source/tests/docs and appear to be generated point-in-time analysis outputs.
- Better stored under an archived reports location (or regenerated on demand) if needed.

**Risk:** Low–Medium.

### 6) Dated local archive snapshot

- `root_cleanup_archive_2026-01-12/`

**Why it should be deleted**
- Dated snapshot directory indicates local archival output, not source of truth.
- Archive copies should live in external backup storage, not active project root.

**Risk:** Medium (confirm external backup retention first).

### 7) Generated audit outputs (regenerable)

- `audit_outputs/` (contains manifests, duplication reports, dead-file reports, etc.)

**Why it should be deleted or pruned**
- Generated by audit tooling and can be recreated.
- Keeping all snapshots in-repo causes noise and larger diffs.

**Risk:** Medium (keep only if trend history is intentionally tracked).

### 8) Runtime alert snapshots (regenerable)

- `alerts/alert_*.json` (historical event files)
- `alerts/alert_history.jsonl` (if historical retention is not required)

**Why they should be deleted or rotated**
- Runtime output artifacts should generally be externalized or gitignored.
- Reduces repository noise and accidental operational data check-ins.

**Risk:** Medium (confirm observability retention policy).

### 9) Agent/archive workspace dumps

- `agent_workspaces/archive/`
- (Optionally) old per-agent workspace dumps under `agent_workspaces/*` that are not active

**Why they should be deleted or moved off-repo**
- Historical coordination artifacts, not product/runtime source.
- Large archive footprints reduce maintainability and blur SSOT for active workflows.

**Risk:** Medium.

### 10) Archived GitHub workflows no longer used

- `.github/workflows/archive/`

**Why it should be deleted**
- Inactive archived workflow copies create CI confusion and duplicate config surface.
- Keep active workflows in `.github/workflows/` only.

**Risk:** Low–Medium.

### 11) Large historical archive tree

- `archive/` (hundreds of files)

**Why it should be deleted or externalized**
- Historical snapshots/reports dominate repository content and are not canonical runtime code.
- Strong candidate for release artifact storage or separate archive repository.

**Risk:** Medium–High (requires coordinated archival policy).

## Tier 3 — **Conditional deletions** (delete only if feature/domain is retired)

### 12) Root-level WordPress/PHP incident artifacts

- `analytics_code.php`
- `freerideinvestor_menu_setup.php`
- `front-page-remote.php`
- `index-wp-standard.php`
- `wp-config-freerideinvestor.com-investigate.php`

**Why they may be deleted**
- No references from primary Python runtime/docs paths.
- These look like migration/incident/support snippets, not part of the core app runtime.

**Delete condition**
- Delete if WordPress operational support is retired or moved under a dedicated `sites/` or `ops/wordpress/` SSOT location.

**Risk:** Medium.

### 13) Duplicate/sensitive key artifacts

- `thea_key.bin`
- `secure/thea_key.bin`

**Why one/both may need deletion**
- Binary key artifacts should not live in git history unless explicitly intended and encrypted under policy.
- Two key files create SSOT ambiguity (which one is authoritative).

**Delete condition**
- Keep only one managed secret source (or none in repo), and remove the other.

**Risk:** High (security-sensitive: rotate/migrate first).

---

## Recommended execution order

1. Delete Tier 1 items completely.
2. Delete Tier 2 items that are clearly regenerable (`audit_outputs`, dated snapshots) after backup confirmation.
3. For Tier 3, decide policy owner (security/ops) before deletion.
4. Add/strengthen `.gitignore` rules so generated artifacts do not re-enter.

## Validation checklist (run before/after deletion)

```bash
# 1) Ensure legacy top-level handlers are not referenced
rg -n "src\.cli\.commands\.(cleanup_handler|start_handler|status_handler|stop_handler)" src tests

# 2) Ensure command router uses canonical handler path
rg -n "from \.handlers\.(status_handler|stop_handler|start_handler|cleanup_handler|validation_handler)" src/cli/commands/command_router.py

# 3) Track what will be deleted
git status --short

# 4) Run baseline tests after each deletion batch
pytest -q
```

## Summary

If your goal is an SSOT-clean repository, the highest-value deletions are:
- backup/merge artifacts,
- duplicate handler copies outside canonical paths,
- generated audit/runtime snapshots in the repo root,
- stale root-level reports not linked by active docs,
- and oversized historical archive trees that should live outside the active code repository.
